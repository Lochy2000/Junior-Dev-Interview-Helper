## How Can I Contribute?

### 🐛 Reporting Bugs or Issues

Found a mistake? See something that could be clearer?

1. Check if the issue already exists in [Issues](https://github.com/Lochy2000/Junior-Dev-Interview-Helper/issues)
2. If not, create a new issue with:
   - Clear title
   - Description of the problem
   - Location (file and line number if applicable)
   - Suggestion for improvement (optional)

### 💡 Suggesting Enhancements

Have an idea for new content or improvements?

1. Check [Issues](https://github.com/Lochy2000/Junior-Dev-Interview-Helper/issues) and [Discussions](https://github.com/Lochy2000/Junior-Dev-Interview-Helper/discussions)
2. Create a new discussion or issue describing:
   - What you'd like to see added/changed
   - Why it would be valuable
   - Any relevant examples or resources

### 📝 Contributing Content

You can contribute:

- **New documentation** - Add guides for new topics
- **Improved explanations** - Make existing content clearer
- **Code examples** - Add practical examples
- **Interview questions** - Share common interview questions
- **Cheat sheets** - Create quick reference guides
- **Diagrams** - Visual aids for complex topics
- **Translations** - Make content accessible in other languages
- **Typo fixes** - Even small fixes matter!

## Getting Started

### 1. Fork the Repository

Click the "Fork" button at the top right of this repository.

### 2. Clone Your Fork

```bash
git clone https://github.com/YOUR-USERNAME/Junior-Dev-Interview-Helper.git
cd Junior-Dev-Interview-Helper
```

### 3. Create a Branch

```bash
git checkout -b feature/your-feature-name
```

**Branch naming conventions:**
- `docs/topic-name` - Documentation updates
- `feature/feature-name` - New features
- `fix/issue-description` - Bug fixes
- `cheatsheet/topic-name` - New cheat sheets

### 4. Make Your Changes

Edit files using your favorite text editor. Follow the [Style Guide](#style-guide) below.

### 5. Test Your Changes

- Check for typos and formatting
- Ensure code examples work
- Verify all links are working
- Preview Markdown rendering if possible

## Contribution Guidelines

### Documentation

- **Be beginner-friendly** - Explain concepts clearly, avoid jargon
- **Use examples** - Show, don't just tell
- **Add context** - Explain why something is important
- **Keep it concise** - Respect the reader's time
- **Include resources** - Link to official docs and learning resources

### Code Examples

- **Test your code** - Ensure examples work as expected
- **Add comments** - Explain what the code does
- **Use best practices** - Show good coding standards
- **Keep it simple** - Examples should be easy to understand

### Cheat Sheets

- **Be comprehensive** - Cover common use cases
- **Use clear formatting** - Easy to scan and reference
- **Include examples** - Show usage, not just syntax
- **Add explanations** - Brief notes for clarity

## Style Guide

### Markdown Formatting

```markdown
# Main Title (H1) - One per file

## Section (H2)

### Subsection (H3)

#### Sub-subsection (H4)

- Use bullet points for lists
- Keep consistent formatting

1. Use numbered lists for steps
2. In sequential order

**Bold** for emphasis
*Italic* for technical terms
`inline code` for code snippets

```language
code blocks with syntax highlighting
```

> Blockquotes for important notes
```

### Code Formatting

- Use appropriate syntax highlighting (```javascript, ```python, ```sql)
- Include comments for complex code
- Keep examples practical and runnable
- Show both good and bad examples when helpful

**Example:**

````markdown
```javascript
// ✅ Good - clear variable names
const userAge = 25;
const isAdult = userAge >= 18;

// ❌ Bad - unclear names
const x = 25;
const y = x >= 18;
```
````

### Language

- **Simple** - Use plain English, avoid jargon
- **Active voice** - "Use map to transform arrays" not "map can be used to transform arrays"
- **Second person** - Address the reader as "you"
- **Inclusive** - Use "they/them" for pronouns, avoid assumptions

### Structure

Each documentation file should include:

1. **Title** with brief description
2. **Table of Contents** for easy navigation
3. **Clear sections** with descriptive headers
4. **Code examples** showing practical usage
5. **Interview questions** (where applicable)
6. **Resources** for further learning
7. **Keywords** at the bottom for SEO

## Commit Message Guidelines

### Format

```
Type: Brief description (50 chars or less)

Detailed explanation if needed (72 chars per line)

- Additional context
- Why the change is needed
```

### Types

- `docs:` - Documentation changes
- `feat:` - New content or features
- `fix:` - Corrections or bug fixes
- `style:` - Formatting, typos (no content change)
- `refactor:` - Reorganizing content
- `chore:` - Maintenance tasks

### Examples

```bash
git commit -m "docs: Add React hooks examples to react.md"

git commit -m "feat: Create TypeScript fundamentals guide"

git commit -m "fix: Correct SQL JOIN syntax in databases.md"

git commit -m "cheatsheet: Add Docker commands reference"
```

## Pull Request Process

### 1. Ensure Quality

Before submitting:

- [ ] Content is accurate and tested
- [ ] Spelling and grammar are correct
- [ ] Code examples work as expected
- [ ] Markdown renders properly
- [ ] Links are functional
- [ ] Follows style guide

### 2. Update Documentation

If you're adding new files:

- [ ] Update main README.md with links
- [ ] Add to table of contents where appropriate
- [ ] Include keywords for searchability

### 3. Submit Pull Request

1. Push your changes to your fork
2. Go to the original repository
3. Click "New Pull Request"
4. Select your branch
5. Fill in the PR template:

```markdown
## Description
Brief description of changes

## Type of Change
- [ ] Documentation update
- [ ] New content
- [ ] Bug fix
- [ ] Cheat sheet

## Checklist
- [ ] Content is accurate
- [ ] Examples are tested
- [ ] Follows style guide
- [ ] Updated README if needed
```

### 4. Review Process

- A maintainer will review your PR
- Address any feedback or requested changes
- Once approved, your PR will be merged!
- You'll be added to the contributors list 🎉
